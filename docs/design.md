# Galgame Notes Assistant — 设计文档

> 状态：设计阶段 | 最后更新：2026-06-28

## 1. 动机与目标

玩 galgame / 视觉小说时内容量大，手动记笔记频繁打断沉浸感。本项目做一个 AI 伴读：

- **实时记录**：游玩时后台抓取屏幕上的重要文本，不用手动停下来写
- **自动整理**：把原始台词归纳成结构化笔记（人物、场景、事件、伏笔、选项分支），玩完一段就有现成笔记
- **剧情对话**：随时能和一个"知道你正在玩的游戏剧情进展"的 AI 提问、讨论；讨论中冒出的结论回填进笔记
- **按游戏分库**：每款游戏独立记忆，互不串味

## 2. 关键决策

| 维度 | 选择 | 理由 |
|---|---|---|
| 主抓取 | Textractor（内存钩子） | 准确、零延迟、带说话人名、中日文通吃 |
| 兜底抓取 | 截屏 + 本地 PaddleOCR | 覆盖模拟器 / 非标引擎 / 加壳作品 |
| 语言 | 中文，无翻译层 | 用户主要玩中文版 |
| AI 角色 | 自动整理 + 对话陪读，共用记忆 | 两者都要 |
| LLM 后端 | 云端 API | 质量稳、长上下文塞台词、不占本机 GPU |
| 笔记存储 | 本地 Markdown，按游戏分文件夹 | 人可读可改、可 git、不锁定 |

## 3. 非目标（v1 不做）

- 自研文本钩子引擎（交给 Textractor）
- 翻译层（暂只服务中文版）
- 语音 / 音频转写
- 多用户、云同步

## 5. 组件细节

### 5.1 抓取层

**Textractor 通道**
- 用户照常用 Textractor 挂游戏进程、必要时选 H-code
- 额外装一个小扩展，把抓到的文本（含说话人名、线程号）经 WebSocket 实时推给核心服务
- 产出事件：`{game_id, source:"textractor", ts, speaker, text, thread_id}`

**OCR 通道**
- 按用户选定的窗口或区域以 ~2–3 Hz 截屏
- 本地 PaddleOCR 识别
- 去重：仅在文本相对上一帧发生变化时推送新台词（解决 OCR 每帧重复同一句的问题）
- 拿不到说话人，留给 AI 从上下文推断
- 产出事件：`{game_id, source:"ocr", ts, speaker:null, text}`

**通道路由**
- 两条通道都产出统一格式的"台词事件"，下游不关心来源
- 用户可在界面手动切换通道；或核心服务检测到 Textractor 无新事件时提示切 OCR

### 5.2 核心服务

**台词缓冲区**
- 每款游戏一份 append-only 的原始 transcript，作为唯一事实源
- 落盘到 `transcript.md`

**自动整理器**
- 触发：每攒够 N 条新台词 或 每隔 M 分钟（批量，控制成本，不是每句都调）
- 调云端 LLM（用较便宜档），把新台词**增量合并**进结构化笔记
- 维护：人物表、场景/章节、关键事件、伏笔、选项分支
- 幂等合并：新整理结果与已有笔记合并，不覆盖人工编辑

**对话引擎**
- 用户提问时，检索相关上下文：近期台词 + 相关笔记 + 人物表
- 调云端 LLM（用更强档）回答
- 讨论中冒出的结论回填进 `notes.md` 的"讨论洞察"区

### 5.3 记忆库（按游戏分库）

每款游戏一个文件夹：

```
memory/<game_slug>/
  meta.json        # 游戏名、抓取模式、设置
  transcript.md    # 原始台词流（append-only，事实源）
  notes.md         # 结构化笔记（AI 维护，人可改）
  characters.md    # 人物表
```

### 5.4 界面

- 与游戏并排显示的侧栏
- 实时台词流（最近 N 条）
- 结构化笔记视图（可编辑）
- 聊天框
- 游戏切换器
- 设置：Textractor 路径、OCR 截取区域、LLM 配置、快捷键

## 6. 数据流

```
游戏
  → Textractor / OCR
  → 统一台词事件流
  → transcript 追加 + 触发自动整理
  → notes.md 增量更新
  → 用户提问时，对话引擎检索 transcript + notes + characters → 云端 LLM → 回答
  → 讨论洞察回填 notes.md
```

## 7. 错误处理

- Textractor 钩不到 → 提示用户切 OCR 通道
- OCR 文本有噪 → 进 LLM 前轻量清洗
- 重复台词（钩子重复推送 / OCR 同帧）→ 按 (text, 时间窗) 去重
- LLM 调用失败 → 保留原始 transcript，稍后重试；**绝不丢失已抓文本**
- 游戏未被识别 → 手动创建游戏 profile

## 8. 待定问题（后续迭代决定）

- 分支 / 线路：是否给 transcript 打 save 槽 / 路线标记？（v1 先做可选手动标记）
- OCR 文本框区域：自动检测 vs 手动框选
- 对话上下文检索：简单近期窗口 vs 对 transcript 做向量检索
- 界面技术：Tauri vs Flask + 浏览器
- 自动整理的"重要性"判定：如何区分关键剧情 vs 日常水字

## 9. 技术栈（拟定）

- Python 3.11 核心服务（常驻进程，FastAPI/WebSocket 服务端）
- Textractor（已有工具）+ 自写 WebSocket 扩展
- PaddleOCR（本地，中日文）
- 云端 LLM，OpenAI 兼容接口（Proma Cloud / DeepSeek / Qwen / Gemini）
- Tauri（Rust + webview）或 Flask + 浏览器
- 本地 Markdown 文件作为记忆库
