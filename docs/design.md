# Galgame Notes Assistant — 设计文档

> 状态：设计完成，待开工 | 最后更新：2026-06-28

## 1. 动机与目标

玩 galgame / 视觉小说时内容量大，手动记笔记频繁打断沉浸感。本项目做一个 AI 伴读：

- **实时记录**：游玩时后台抓取屏幕上的重要文本，不用手动停下来写
- **自动整理**：把原始台词归纳成结构化笔记（人物、场景、事件、伏笔、选项分支），玩完一段就有现成笔记
- **剧情对话**：随时能和一个"知道你正在玩的游戏剧情进展"的 AI 提问、讨论；讨论中冒出的结论回填进笔记
- **按游戏分库**：每款游戏独立记忆，互不串味

## 2. 关键决策

| 维度 | 选择 | 理由 |
|---|---|---|
| 主抓取 | Textractor（内存钩子） | 准确、零延迟、带说话人名、中日文通吃 |
| 兜底抓取 | 截屏 + 本地 PaddleOCR | 覆盖模拟器 / 非标引擎 / 加壳作品 |
| 语言 | 中文，无翻译层 | 用户主要玩中文版 |
| AI 角色 | 自动整理 + 对话陪读，共用记忆 | 两者都要 |
| LLM 后端 | 云端 API | 质量稳、长上下文塞台词、不占本机 GPU |
| 笔记存储 | 本地 Markdown，按游戏分文件夹 | 人可读可改、可 git、不锁定 |
| 剧情检索 | 分阶段：v1 近期窗口+笔记全文，后续向量检索 | 先快出原型，留升级路径 |
| 界面 | Flask + 浏览器（v1），后续可迁 Tauri | 一套 Python 最快出原型 |
| OCR 区域 | 手动框选为主，后续加自动建议 | 可靠优先，框选位置存 meta.json |
| 多线路 | 默认按时间追加 + 手动路标按钮 | 不强加结构，留组织能力 |
| 重要性判定 | 全交给 LLM + 标准 prompt，transcript 全量兜底 | galgame 千差万别，硬规则会误杀 |

## 3. 非目标（v1 不做）

- 自研文本钩子引擎（交给 Textractor）
- 翻译层（暂只服务中文版）
- 语音 / 音频转写
- 多用户、云同步
- 向量检索（留到 v2）
- 自动线路推断（留到 v2）

## 5. 组件细节

### 5.1 抓取层

**Textractor 通道**
- 用户照常用 Textractor 挂游戏进程、必要时选 H-code
- 额外装一个小扩展，把抓到的文本（含说话人名、线程号）经 WebSocket 实时推给核心服务
- 产出事件：`{game_id, source:"textractor", ts, speaker, text, thread_id}`

**OCR 通道**
- 按用户手动框选的台词区域（存于 `meta.json`，下次玩同一款自动恢复）以 ~2–3 Hz 截屏
- 本地 PaddleOCR 识别该区域
- 去重：仅在文本相对上一帧发生变化时推送新台词（解决 OCR 每帧重复同一句的问题）
- 拿不到说话人，留给 AI 从上下文推断
- 产出事件：`{game_id, source:"ocr", ts, speaker:null, text}`
- **区域定法**：v1 手动框选为主（可靠）；后续加"自动建议区域、用户确认"的便利按钮，不做纯自动检测（galgame UI 千差万别，纯自动易找错）

### 5.2 核心服务

**台词缓冲区**
- 每款游戏一份 append-only 的原始 transcript，作为唯一事实源
- 落盘到 `transcript.md`，**全量保留、一字不丢**（重要性判定错误的兜底）

**自动整理器**
- 触发：每攒够 N 条新台词 或 每隔 M 分钟（批量，控制成本，不是每句都调）
- 调云端 LLM（用较便宜档），把新台词**增量合并**进结构化笔记
- 维护：人物表、场景/章节、关键事件、伏笔、选项分支
- 幂等合并：新整理结果与已有笔记合并，不覆盖人工编辑
- **重要性判定**：全交给 LLM，在 prompt 里写死标准——新人物首次出场 / 伏笔暗示 / 剧情转折 / 关键选项 → 重要；日常寒暄 / 重复 / 纯环境描写 → 不重要。不用硬规则（galgame 多样性高，规则会误杀）。判断错了也不丢信息，因 transcript 全量保留。

**对话引擎**
- v1 检索策略：**近期窗口 + 笔记全文**。把最近 N 条原始台词 + 整份 `notes.md`（已浓缩、体积小）+ `characters.md` 拼成上下文给 LLM。覆盖 80% 提问。
- 调云端 LLM（用更强档）回答
- 讨论中冒出的结论回填进 `notes.md` 的"讨论洞察"区
- **后续升级（v2）**：加向量检索，对全量 transcript 建 embedding 索引，支持"几千句之前的伏笔"这类长程精准捞取

### 5.3 记忆库（按游戏分库）

每款游戏一个文件夹：

```
memory/<game_slug>/
  meta.json        # 游戏名、抓取模式、OCR 框选区域、设置
  transcript.md    # 原始台词流（append-only，事实源，全量保留）
  notes.md         # 结构化笔记（AI 维护，人可改）
  characters.md    # 人物表
```

### 5.4 多线路 / 多周目处理

- **默认**：所有台词按游玩时间顺序追加进 transcript，不强加分线结构
- **手动路标**：界面提供"打路标 / 章节标记"按钮，开始玩某条线路或某章节时点一下，transcript 里插入标记（如 `═══ B 线开始 ═══`），讨论时 AI 能据此识别分段
- 不做自动线路推断（v1），等核心跑通且有强需求再考虑

### 5.5 界面

- **技术**：Flask 本地网页，浏览器打开（可设为"应用模式"看起来像独立窗口）；后续可迁移到 Tauri，前端代码基本可复用
- 与游戏并排显示的侧栏
- 实时台词流（最近 N 条）
- 结构化笔记视图（可编辑）
- 聊天框
- 游戏切换器
- 路标 / 章节标记按钮
- 设置：Textractor 路径、OCR 框选区域、LLM 配置、快捷键

## 6. 数据流

```
游戏
  → Textractor / OCR
  → 统一台词事件流
  → transcript 追加 + 触发自动整理
  → notes.md 增量更新
  → 用户提问时，对话引擎拼近期台词 + 笔记 + 人物表 → 云端 LLM → 回答
  → 讨论洞察回填 notes.md
```

## 7. 错误处理

- Textractor 钩不到 → 提示用户切 OCR 通道
- OCR 文本有噪 → 进 LLM 前轻量清洗
- 重复台词（钩子重复推送 / OCR 同帧）→ 按 (text, 时间窗) 去重
- LLM 调用失败 → 保留原始 transcript，稍后重试；**绝不丢失已抓文本**
- 游戏未被识别 → 手动创建游戏 profile
- OCR 框选偏移（台词框位置变了）→ 用户重新框选，更新 meta.json

## 8. 已定决策汇总（原开放问题）

1. **剧情检索** → 分阶段：v1 近期窗口 + 笔记全文；v2 加向量检索
2. **界面技术** → Flask + 浏览器（v1），后续可迁 Tauri
3. **OCR 区域** → 手动框选为主 + 后续加自动建议，位置存 meta.json
4. **多线路** → 默认按时间追加 + 手动路标按钮，不做自动推断
5. **重要性判定** → 全交给 LLM + 标准 prompt，transcript 全量保留兜底

## 9. 技术栈

- Python 3.11 核心服务（常驻进程，Flask + WebSocket）
- Textractor（已有工具）+ 自写 WebSocket 扩展
- PaddleOCR（本地，中日文）
- 云端 LLM，OpenAI 兼容接口（Proma Cloud / DeepSeek / Qwen / Gemini）
- Flask + 浏览器作为界面（v1）
- 本地 Markdown 文件作为记忆库
